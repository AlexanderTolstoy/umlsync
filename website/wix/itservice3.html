​ Project hosting providers:  Nowday there are a lot of web-based project hosting providers: GitHub, GoogleCode, SourceForge, BitBucket etc. You could choose between public and private repository usage. Most of these storage services provide a developer API for 3pp aplications creation. UMLSync is based on such APIs which allow us to do not store user data on our server and keep user documentation togather with code.

 

Web collaboration:  It looks really comfortable to send the online reference on diagrams and documents to your collegues. You could create, modify and commit documents into your github repository and control an access rights via GitHub.
Snipets - is one more options for online collaboration: you could summarize your knowledge about feature and share them with your collegues as a sequence of coments over the online documents, diagrams and code;

 

Local clone of repo:  The major problem of web based tools is offline mode. Therefore most of web-tools are transition of existing PC products [like MS Office].
UMLSync provides a WebSync plug-in for the Eclipse IDE and open WebSync HTTP interface description for developers. It is possible to implement custom plug-in for any IDE.
WebSync interface was extended with indexer API for semi-automated diagram creation. It is similar to Eclipse Dstore API for remote development.

​

This project was found in 2005 as my post graduate theme in the Saint-Petersburg State University (Russia). It was an idea to create simi-automated framework for UML diagrams creation.
​
I did not thought about web-based solutoin or web-based project hosting providers. The result of this project is [googlecode link]. It was a simple idea to create diagrams by name-based and relation-based rules and use these rules for the diagram update in the future. That project was based on the Visual-Paradigm open API and as a result it was blocked by Visual Paradigm limitations on DND: [link to the request]. But source code is partially re-used for a WebSync project.


In 2010 I have met with jQuery UI library and find out that it is quit easy to create UML Class element with it and make this element draggable, resizable, editable and sortable etc.

 

 

Today I have a lot of functionality implemented:

​

1. Four diagram are stable, but I have 16 diagrams protoyped
2. Diagrams integration into markdown
3. Wiki-like navigation between diagrams and documents
4. Framework for GitHub with multiple editor
6. Key handler and operation manager [Ctrl-Z/Y/C/V]
7. Prototyped Selenium based test suit. Mainly for opration manager testing.
8. Plugin for Eclipse to provide an eccess to the local clone of repo + indexer

Features

How to minimize your efforts for internal documentation creation and support ?

​

The main value of UMLsync project is to make UML diagrams more usable. Thanks to HTML5 you could create, edit and share UML diagrams in any modern browser. You don't have to install any software on your PC and easily share diagrams and documents with collegues. You could easily integrate diagrams into markdown documentation and use cross-references for navigarion through the project documentation. And if you want to deal with local clone of repository then you could setup WebSync plug-in for Eclipse which provide an access to the project hierarchy and content.

Two minutes for​

a UML diagram creation

Most of the developers do not create UML diagrams because it takes a lot of time. According to our statistics a minimum time for a UML diagram creation is 10 minutes (for existing code). It looks really boring especially when you have so nice tools like Eclipse or NetBeans. So, WebSync plug-in for Eclipse allows you to extract and filter information about source code and use it for diagrams creation. UmlSync + WebSync do not provide an automated diagrams creation because in the most cases they are useless.

Semi-automated approach allows you to create usefull diagrams in a minimal time.

Integrate UML diagrams​

into online docs​

How often do you have to keep 2 instances of diagram: one in XMI or any other custom format and another one in PNG/JPG/SVG ? Today, thanks to GitHub all developer could create an online documentation which located in the same repository as code. It means that each branch or revision could contain a valid documentation, API, installation guide etc.

So, the next logical step is to integrate diagrams into markdown documentation !!!

Access to the local​

clone of your repository

WebSync is an open HTTP interface for synchronization of web applicaiton with client's file system through the source code editor applications like Eclipse, NetBeans, SourceNavigator etc. It is quite similar to the Eclipse DStore (see "RSE DStore Developer Guide" ) except SSL support and provides the following functionality:

1. File tree naviagration
2. File hierarchy managment
3. Content managment
4. Source code index database access

WebSync is an open HTTP interface with a simple secret key-based security.

Snippets - the way​

to share knowledge

Snippet was intended to improve the process of communication with colleagues.
Snippet is the sequence of custom comments along with documentation, diagrams and source code which can be shared with team and saved to the knowledgebase of your project.
All you have to do is click on the necessary element and add your comment. Then switch to the next document or diagram and simply add your next comment.
So, that means you can now share your own comments as well as diagrams and documentation with fellow team members and save further edits to the project knowledge database.
It is essential to use snippets in case of outsourcing or multi-site development.

Cross-references for​

diagrams and docs​

The one of the major feature of UmlSync is cross-references support by each element of diagram. You could create one or more references for each element and all these references will be available for embedded diagrams too.
For example: You could insert the common platform components overview diagram on the first README.md document. User could click on the concreate UML-component of embedded diagram and switch to the component description page by reference etc.
Thus you could make a smooth transition from a High Level Design of your project to a Low Level Design of concreate module, component or feature.
